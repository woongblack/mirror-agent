# Mirror Agent — Project Roadmap

> 각 단계는 **독립적 가치**를 갖도록 설계되었다.  
> 어느 단계에서 멈춰도 그 자체로 포트폴리오 자산이 된다.  
> 
> **작성일:** 2026-04-19  
> **원칙:** Harness 95% / Execution 5%

---

## 로드맵 개요

```
Phase 0 ── 문서화 완료 ───────── [이미 완료된 상태]
   ↓
Phase 1 ── 레포 초기화 + 공개 ── [다음 작업]
   ↓
Phase 2 ── Re-Applicator v0.1 ── [첫 동작하는 버전]
   ↓
Phase 3 ── 실사용 + 개선 ─────── [도구로서의 검증]
   ↓
Phase 4 ── 자동화 확장 ───────── [Extractor/Generalizer]
   ↓
Phase 5 ── 다중 에이전트 통합 ── [Socratic/Contrarian 결합]
   ↓
Phase 6 ── 외부 연동 ─────────── [Jira/Confluence]
```

**각 Phase의 "멈춤 가능 지점"이 명확하다.** 어디서 멈춰도 그 지점까지의 산출물은 완성된 상태로 남는다.

## Phase 0 — 문서화 (완료)

**목적:** 프로젝트의 존재 이유와 설계 원칙을 언어화한다.

### 산출물
- [x] `docs/origin/conversation-log.md` — 프로젝트 기원 대화 로그
- [x] `README.md` — 프로젝트 선언문
- [x] `docs/architecture/historical-agent-plan.md` — 핵심 파이프라인 설계
- [x] `data/rules/manual-v0.1/rules.md` — 수동 규칙 10개 초안

### 이 단계가 갖는 독립적 가치
- 자기 객관성 문제에 대한 1차 성찰 완료
- 프로젝트를 말로 설명할 수 있게 됨 (면접 서사 가능)
- **코드 없이도 이미 포트폴리오 자산**

## Phase 1 — 레포 초기화 + 공개

**목적:** 문서를 공개 레포로 공개하고, ALLBLUE의 포지션을 재정의한다.

### 할 일
- [ ] GitHub 레포 `mirror-agent` 생성
- [ ] Phase 0 산출물 배치
  ```
  mirror-agent/
  ├── README.md
  ├── docs/
  │   ├── origin/conversation-log.md
  │   └── architecture/historical-agent-plan.md
  └── data/rules/manual-v0.1/rules.md
  ```
- [ ] ALLBLUE README 상단에 "프로젝트 성격 고지" 단락 추가
- [ ] Mirror Agent에서 ALLBLUE 레포 링크 참조
- [ ] (선택) 개인 블로그 또는 LinkedIn에 프로젝트 소개 짧은 글

### 이 단계가 갖는 독립적 가치
- 포트폴리오에 **"1인 개발자의 자기 객관성"** 이라는 독창적 주제 등장
- ALLBLUE가 **기술 실습 + 메타 도구 테스트베드**로 재포지셔닝됨
- **코드 한 줄 없이도 면접 서사 완성**

### 검증 질문
- 외부 사람이 이 레포를 10분 읽고 "이 사람이 뭘 하려는지" 이해할 수 있는가?
- ALLBLUE 개선을 위한 도구인지, 범용 도구인지 포지션이 명확한가?

### 예상 소요
1~2일

## Phase 2 — Re-Applicator v0.1 (첫 동작 버전)

**목적:** 수동 규칙 8개로 ALLBLUE README를 검토하는 최소 동작 버전.

### 할 일
- [ ] Python 프로젝트 초기화 (uv 또는 poetry)
- [ ] 규칙 JSON 파싱 로더
- [ ] 문서 분석기 (마크다운 → 메타데이터 추출)
- [ ] 규칙 매칭 로직 (trigger_conditions 기반)
- [ ] 질문 생성기 (template 치환 + LLM 정제)
- [ ] 리포트 생성기 (마크다운 출력)
- [ ] **방어 예측 로직** ⭐ (Mirror Agent 고유 가치)
- [ ] CLI 인터페이스: `mirror review <document_path>`

### 이 단계가 갖는 독립적 가치
- 동작하는 프로토타입
- ALLBLUE README에 대한 실제 비판 리포트 산출
- **"자기 프로젝트를 자기 도구로 비판한 증거"**

### v0.1 성공 기준 (historical-agent-plan.md에서 확정됨)
- ALLBLUE README 적용 시 5개 기대 비판 중 **3개 이상 생성**
- 방어 예측이 최소 2개 비판에 포함됨
- 리포트가 일반 LLM 비판("사용자 검증이 필요합니다")과 구분됨

### 실패 시 대응
- 수동 규칙 자체의 질 재검토 → rules.md 업데이트
- 규칙 매칭 로직의 엄격도 조정
- 템플릿 치환 로직 디버깅

### 예상 소요
1~2주

## Phase 3 — 실사용 + 개선 사이클

**목적:** 도구를 실제로 쓰면서 개선한다. **이 단계가 이 프로젝트의 진짜 검증이다.**

### 할 일
- [ ] ALLBLUE README 개선 → 리포트 재생성 → 수정 사이클 N회 돌리기
- [ ] 매 사이클의 "수용한 비판 / 반려한 비판" 기록
- [ ] 반려 이유를 `data/rejections.md`에 축적 (Generalizer 자동화의 재료)
- [ ] 방어 패턴이 발견되면 `data/defense-patterns.json`에 추가
- [ ] 수동 규칙 10번 슬롯에 새 규칙 채우기 (Rule 10)
- [ ] 다른 문서(이전 프로젝트 PRD, 기획안)에도 적용해보기
- [ ] 사용 로그를 `docs/case-studies/`에 정리

### 이 단계가 갖는 독립적 가치 — **가장 중요**
- **"실제로 쓴 도구"라는 증거** (GitHub 커밋 타임스탬프로 증명됨)
- 포트폴리오 인터뷰 질문 대응 가능: "이거 실제로 쓰세요?" → "네, 이 커밋 보세요"
- 도구의 한계와 개선 방향에 대한 구체적 이해

### 검증 질문
- 이 도구를 **계속 쓰고 싶은가**? 아니면 만들어놓고 안 쓰게 되는가?
- 에이전트 비판을 **실제로 받아들였는가**? 아니면 방어 모드로 전환됐는가?

### ⚠️ 이 단계가 프로젝트의 진짜 실패 조건
README에 이미 명시됨:
> "내가 이 도구를 실제로 매주 쓰지 않으면 실패다."

### 예상 소요
2~4주 (사이클 횟수에 따라)

## Phase 4 — Extractor + Generalizer 자동화

**목적:** 수동 규칙 작성을 자동화한다.

### 할 일
- [ ] **Stage 1 Extractor 구현**
  - [ ] 과거 대화/문서에서 Critique Unit 자동 추출
  - [ ] conversation-log.md를 정답셋으로 재현율 측정
  - [ ] 새 대화 로그를 넣었을 때 신규 비판 추출 확인
- [ ] **Stage 2 Generalizer 구현**
  - [ ] Critique Unit 클러스터링
  - [ ] 추상 규칙 생성
  - [ ] GitHub PR 자동 생성 워크플로우 (`.github/workflows/rule-review.yml`)
  - [ ] 수동 규칙과 자동 생성 규칙의 품질 비교
- [ ] 자동 생성 규칙의 승인율 추적 (품질 지표)

### 이 단계가 갖는 독립적 가치
- 완전 자동화된 3-Stage 파이프라인
- 규칙 진화가 Git 히스토리로 가시화됨
- **"LLM 기반 메타 학습 시스템"** 이라는 기술적 주장 가능

### 검증 질문
- Generalizer 자동 생성 규칙의 PR 승인율이 50% 이상인가?
- 수동 규칙보다 자동 생성 규칙이 더 정확한가, 덜 정확한가?

### ⚠️ 실패 시 대응
historical-agent-plan.md의 솔직한 경고:
> "이 플랜대로 가면 Generalizer 단계에서 무조건 벽에 부딪힐 거예요."

실패해도 괜찮다. v0.1의 수동 규칙 방식이 이미 동작하므로, 이 단계 실패 자체가 **"LLM의 일반화 한계에 대한 실증 케이스"** 라는 포트폴리오 가치를 가진다.

### 예상 소요
2~3주

## Phase 5 — 다중 에이전트 통합

**목적:** Historical Agent를 Socratic/Contrarian Agent와 결합한다.

### 할 일
- [ ] **Socratic Agent 구현**
  - [ ] "왜?"를 5번 다른 각도로 묻는 LLM 체인
  - [ ] 가정 드러내기 패턴
- [ ] **Contrarian Agent 구현**
  - [ ] 반대 전제 탐색
  - [ ] 당연한 가정에 회의 제기
- [ ] **오케스트레이션 레이어**
  - [ ] 3개 에이전트 병렬 또는 순차 실행
  - [ ] 결과 통합 및 우선순위 결정
  - [ ] Historical Agent가 다른 에이전트의 출력을 "과거 궤적 기준으로 재검증"하는 뽀짝이/뽀야 구조
- [ ] (선택) LangGraph 도입 검토 (상태가 복잡해지면)

### 이 단계가 갖는 독립적 가치
- 완전체 Mirror Agent 팀
- 뽀짝이/뽀야 구조의 확장 버전 (개인화 레이어 추가)
- 다른 1인 개발자에게 오픈 소스로 공개 가능

### 검증 질문
- 3개 에이전트 결합이 Historical Agent 단독보다 **의미 있게 나은가**?
- 아니면 에이전트를 늘린 만큼 노이즈만 늘었는가? (이 경우 단독 유지)

### 예상 소요
2~3주

## Phase 6 — 외부 연동

**목적:** Jira/Confluence와 통합하여 실제 업무 흐름에 녹인다.

### 할 일
- [ ] **Atlassian MCP 연동**
  - [ ] Confluence 페이지를 Critique Unit 입력 소스로 사용
  - [ ] 확정된 비판 리포트를 Confluence에 자동 저장
  - [ ] Jira 티켓 자동 생성 (비판 → 액션 아이템)
- [ ] **Claude Code sub-agent 호출**
  - [ ] 확정된 기획 → Claude Code에게 넘기는 프롬프트 생성
  - [ ] 구현 완료 후 Jira 티켓 자동 업데이트
- [ ] 회사에서 실제 사용 시나리오 설계

### 이 단계가 갖는 독립적 가치
- "Mirror Agent → Claude Code → Jira/Confluence" 전체 파이프라인
- 카페24 JD의 모든 키워드가 **자연스럽게** 커버됨 (체크리스트 느낌 없이)

### 주의
이 단계는 **취업 이후 실제 회사 업무에 적용하면서 하는 게 자연스럽다**. 개인 프로젝트로 강제할 필요 없다.

### 예상 소요
1~2주

## 멈춤 가능 지점 매트릭스

각 Phase 종료 시 프로젝트를 멈춰도 **그 시점까지의 가치가 남는다.**

| Phase 종료 시점 | 남는 가치 | 포트폴리오 강도 |
| --- | --- | --- |
| Phase 0 | 프로젝트 컨셉 + 설계 문서 | ★★☆☆☆ |
| Phase 1 | 공개 레포 + ALLBLUE 재포지셔닝 | ★★★☆☆ |
| Phase 2 | 동작하는 v0.1 도구 | ★★★★☆ |
| **Phase 3** | **실사용 증거 + 자기 프로젝트 개선 케이스** | **★★★★★** |
| Phase 4 | 자동화 파이프라인 | ★★★★★ |
| Phase 5 | 완전체 에이전트 팀 | ★★★★★ |
| Phase 6 | 엔터프라이즈 연동 | ★★★★★+ |

**가장 중요한 점:** Phase 3까지만 해도 포트폴리오로는 완성이다. Phase 4 이후는 **"이미 완성된 것의 확장"** 이므로 취업 타이밍이나 관심 변화에 따라 유연하게 판단하면 된다.

---

## 우선순위 추천

**취업 지원 직전까지라면:**
- Phase 1 → 2 → 3 까지 (2~4주) 완주
- Phase 4~6는 "로드맵으로 제시" 상태로 두기

**취업 합격 후에도 계속할 거라면:**
- Phase 4~5는 실제 업무 중 틈틈이
- Phase 6는 회사 환경에서 자연스럽게

**이 프로젝트를 완전히 다른 방향으로 틀게 된다면:**
- Phase 3까지의 결과물을 **"중단된 프로젝트"** 가 아닌 **"1차 검증 완료 프로젝트"** 로 기록

## 각 Phase 착수 전 체크리스트

Phase 진입 전 스스로 묻기:

- [ ] 이전 Phase의 **실사용 피드백**을 반영했는가?
- [ ] 새 Phase가 **정말 필요한 이유**가 있는가, 아니면 "있어야 할 것 같아서" 진행하는가? (Rule 5 자기 적용)
- [ ] 중간에 멈춰도 **이 Phase까지의 산출물이 독립적 가치를 가지는가**?

**마지막 질문이 가장 중요하다.** "완주해야만 의미 있는 프로젝트"는 실패 확률이 높다.

---

## 이 로드맵 자체의 한계

정직하게 기록해둔다:

1. **예상 소요 시간은 낙관적**일 수 있다. 실제로는 1.5배~2배 걸린다고 가정하는 게 안전함.
2. **Phase 간 의존성**을 낮게 설계했지만, Phase 2~3의 실제 결과가 Phase 4~5 설계를 뒤집을 수 있다.
3. **이 로드맵 자체가 자기 합리화의 결과일 수 있다** — "로드맵을 만들었으니 진행해야 한다"는 함정에 빠지지 않도록 주의. Phase 3 실사용에서 도구의 실제 가치가 의심되면 이 로드맵 전체를 버릴 용기도 필요하다.
