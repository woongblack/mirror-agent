# Mirror Agent — Origin Conversation Log

> 이 문서는 Mirror Agent 프로젝트가 탄생한 대화의 전체 기록이다.
> Historical Agent의 첫 번째 학습 데이터이자, 이 프로젝트의 존재 이유에 대한 1차 자료다.

**일자:** 2026년 4월 19일  
**참여자:** 정재웅 (AI Product Engineer, ALLBLUE 1인 개발자) / Claude (Anthropic)  
**목적:** 카페24 지원용 포트폴리오 프로젝트 설계 → 실제 자기 객관성 문제로 전환

---

## 메타 관찰 (Historical Agent 학습용 태그)

이 대화에서 주목할 패턴:

- **초기 의도와 최종 결론의 격차**: 처음엔 "자율 검수 에이전트" 아키텍처 평가 요청 → 최종적으로 "1인 개발자의 자기 객관성 상실" 문제로 전환
- **방어 → 인정의 전환점**: "좋은 사업 아이디어까지는 아니야"라는 발화
- **자기 비판의 비대칭**: 타인에게 던진 질문을 자신에게는 못 던지는 패턴
- **주요 인정 순간**: 객관성 하락 인정, 프로젝트 성격 재정의 수용

### Claude의 재포지셔닝

이미지 검수용이 아니라, **개발자 본인의 업무 자체를 보조하는 에이전트 팀**으로 재해석.

**제시한 4-레이어 구조:**
- Problem Layer (토론형)
- Architecture Layer (설계자)
- Implementation Layer (Claude Code 활용)
- Verification Layer (검증자)

**이 단계에서 Claude가 제안한 재고 사항:**
- 토론은 Problem Layer에서만
- 아키텍처 설계 에이전트는 "자동화"가 아니라 "의사결정 보조"로
- 코드 레벨은 Claude Code를 sub-agent로 호출하는 게 현실적
- Problem Layer 하나만 먼저 만들어 검증할 것

## Phase 8 — 최종 인정 (프로젝트의 진짜 기원)

### 사용자 입력 (가장 중요)

> "계속하고 싶은 이유는 나의 생각을 실제로 구현하고 싶어서였어. 내가 생각한 대로 바로바로 할 수 있는 자유로움이 있어서. 팀에 있을 때는 이 부분을 자유롭게 하지 못했었거든, 내가 생각한 대로 바로바로 테스트하거나 기능을 추가하거나 하는?
> 
> 너가 얘기한 대로 B플랜이 좋아 보여. **나 스스로의 객관성이 혼자 하다 보니 많이 떨어진 것 같아. 이 부분 인정할게.**
> 
> 솔직히 그리고 좋은 사업 아이디어까지는 아니야. 하나의 프로젝트 정도로 생각하는 거지."

### 이 발화의 의미

이 대화 전체에서 **가장 귀한 데이터**다. 이유:

1. **진짜 동기의 발화**: ALLBLUE는 "좋은 사업 아이디어"가 아니라 "혼자 실행할 자유를 얻기 위한 수단"이었다는 인정
2. **객관성 상실의 자기 진단**: 외부에서 지적받기 전에 스스로 인정
3. **프로젝트 성격의 재정의 수용**: "런칭할 제품"이 아닌 "프로젝트 정도"

**이 문장들이 Mirror Agent의 존재 이유를 완성시킨다.**

## Phase 9 — 프로젝트 최종 포지셔닝

### Claude의 최종 제안

**세 프로젝트의 통합 서사:**

```
[DEKK]
   ↓ 집단적 객관성 상실 (좆목질, 방어 모드)
   ↓
[ALLBLUE]
   ↓ 개인적 실행 자유 추구 → 개인적 객관성 상실 발견
   ↓
[Mirror Agent]
   → 개인적 객관성을 시스템으로 복구하는 메타 도구
```

**ALLBLUE의 새 포지션:** "런칭할 제품"이 아니라 "기술 탐구 + Mirror Agent의 첫 테스트 베드"

**Mirror Agent의 핵심 차별점:**
- Socratic Agent, Contrarian Agent (기본)
- **Historical Agent** (고유 기능) — 사용자가 과거에 타인에게 던진 비판을 현재 문서에 기계적으로 재적용

**면접 서사 최종 버전:**

> "저는 DEKK 팀 프로젝트에서 'Snap 클론 아니냐', '셀러는 어떻게 확보할 거냐' 같은 질문을 던졌지만 팀이 답하지 못했습니다. 팀에서 나와 ALLBLUE를 1인 개발로 시작했는데, Spring Boot와 Redis Streams 기반으로 AI 오케스트레이션 파이프라인을 구현하는 과정에서 제가 DEKK 팀에게 했던 바로 그 질문을 저 자신에게는 한 번도 던지지 않았다는 걸 깨달았습니다.
> 
> 혼자서는 자기 객관성을 유지하기 어렵다는 게 제 결론이었고, 그래서 Mirror Agent Team을 만들었습니다. 핵심은 Socratic Agent와 Contrarian Agent가 아니라, Historical Agent입니다. 제가 과거에 타인에게 던진 비판을 현재 제 문서에 기계적으로 재적용하는 에이전트입니다.
> 
> 첫 테스트 케이스로 제 ALLBLUE README를 넣었더니, 에이전트가 'Snap 클론 아닌가?'라는 질문을 다시 꺼냈습니다. 제가 DEKK 팀에게 했던 바로 그 질문이었습니다. 이 결과를 받아들여 ALLBLUE의 상업적 방향을 보류하고 기술 실습 + 에이전트 테스트 베드로 포지션을 바꿨습니다."

## Phase 10 — 핵심 기술 인사이트

### 사용자의 결정적 질문

> "이 결과를 내기까지 내가 너한테 다양한 내용들을 얘기했는데 이 맥락을 얘기하지 않아도 바로 적용하는 에이전트를 만드는 게 핵심인 거 같은데 맞아?"

### Claude의 답변 요약

**맞다.** 이 프로젝트의 기술적 심장이다.

**왜 단순 RAG로는 부족한가:**

- **표면 패턴 (RAG 가능):** "DEKK에서 셀러 얘기했음"
- **메타 패턴 (진짜 필요):** "당신은 플랫폼 프로젝트를 볼 때 항상 공급자 확보 전략을 먼저 묻는다"

두 번째 레벨의 추론이 있어야 "ALLBLUE에 카페24 셀러 확보 전략이 있는가?"라는 질문이 자동으로 나온다.

**필요한 3단계 파이프라인:**
1. 비판 패턴 추출 (이 대화 같은 궤적에서)
2. 일반화 (구체 사례 → 추상 패턴)
3. 재적용 (새 문서에 기계적으로)

## Historical Agent 학습용 추출 패턴

이 대화에서 자동 추출 가능한 정재웅의 사고 패턴:

### 패턴 A — 플랫폼/마켓플레이스 검토 시
- 공급자 확보 전략 질문 ("셀러 어떻게 구할 건데?")
- 차별화 지점 질문 ("클론이면 똑같아지는 거 아니냐?")
- 이후 단계 질문 ("그래서 이후에 어떻게 할 건데?")

### 패턴 B — 기술 스택 검토 시 (추정, 추가 확인 필요)
- 운영 가능성 질문 (1인이 유지보수 가능한가?)
- Premature optimization 경계 (Phase 1 검증 전에 Phase 3 설계?)

### 패턴 C — 자기 합리화 탐지 언어
- "~라고 생각했어"
- "~라면"
- "~할 수 있을 거라고"
- "기술적으로 어렵지만"

### 패턴 D — 방어 → 인정의 전환점 마커
- "이 부분 인정할게"
- "솔직히 ~까지는 아니야"
- "하나의 ~정도로 생각하는 거지"
